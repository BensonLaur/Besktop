# 渲染性能基线与回归规范

本文档记录 Besktop 全屏桌面舞台的可复现性能基线、测量方法和后续动作开发的回归要求。性能数字只代表对应测试环境，不应直接当作所有设备的帧率承诺；它们的主要用途是在同一环境中比较改动前后差异。

## 基线版本

- 基准日期：2026-07-11。
- 优化前提交：`0418f3f`。
- 第一轮优化提交：`f6b8b7f`。
- 构建：Release x64。
- 舞台：主显示器 `3200 × 2000`，200% DPI。
- 桌面演员：全量 47 个；对照组 10 个。
- 普通阴影和图标平面调试关闭。
- 每组至少运行 30 秒，稳定阶段以进入 `wandering` 后的连续统计为主；启动最差帧单独记录。

## 第一轮优化结果

| 场景 | 指标 | 优化前 | 优化后 |
| --- | --- | ---: | ---: |
| 全量 47 个演员 | 稳定 FPS | 13.27 | 31.0–32.0 |
| 全量 47 个演员 | 平均绘制耗时 | 75.34 ms | 约 31.0–31.9 ms |
| 全量 47 个演员 | 启动最差帧 | 234 ms | 146.9 ms |
| 10 个演员 | 稳定 FPS | 32.8 | 39–40 |
| 10 个演员 | 平均绘制耗时 | 30.39 ms | 约 11.1–11.2 ms |

全量演员稳定帧率提升约 2.4 倍，平均绘制耗时下降约 58%。10 个演员的绘制耗时下降约 63%。`paint avg` 只计算实际绘制时间，`paint fps` 和 `timer fps` 还会受到 `WM_TIMER` 调度与消息循环节奏影响，因此不能简单使用 `1000 / paint avg` 推导最终 FPS。

本轮采用的主要优化：

- 持久复用全屏 HDC/HBITMAP，窗口尺寸变化时才重建。
- 将底色、壁纸和静态任务栏预合成为静态背景，稳态帧只进行背景复制。
- 每个演员每帧只计算一次姿态、身体投影和骨架。
- 将每段四肢的两次线段绘制合并为一次 `DrawLines`。
- 预渲染桌面文字，保留描边、换行、省略号、抖动和淡出。
- 图标主体由 `HighQualityBicubic` 调整为经过截图检查的 `Bilinear` 插值。

## 稳定阶段耗时分布

以下数据来自全量 47 个演员的优化后稳定阶段：

| 阶段 | 平均耗时 |
| --- | ---: |
| 缓冲准备 | 约 0 ms |
| 静态背景复制 | 2.4–2.6 ms |
| 壁纸与任务栏 | 稳态 0 ms，仅首次预合成 |
| 姿态计算 | 约 0.01 ms |
| 投影和骨架准备 | 约 0.02 ms |
| 四肢绘制 | 约 5.8 ms |
| 图标主体绘制 | 20.1–20.5 ms |
| 文字绘制 | 淡出后 0 ms |
| 最终 BitBlt | 2.2–2.5 ms |

当前最大的剩余热点是 GDI+ 图标主体绘制，约占稳定帧总耗时的三分之二。继续增加动作数学计算通常不是主要风险；增加图标层、透明特效、重复缩放、阴影或额外全屏合成更容易造成明显回退。

若目标提高到稳定 50–60 FPS，应先验证能否减少图标主体的 GDI+ 绘制成本。低风险微优化收益耗尽后，再评估 Direct2D/Direct3D 纹理缓存和批处理，不应在缺少数据时直接重写渲染架构。

## 复现方法

Debug 构建可直接启用统计。Release 构建必须显式打开诊断总开关：

```powershell
$env:BESKTOP_ENABLE_DIAGNOSTICS='1'
$env:BESKTOP_FRAME_STATS='1'
$env:BESKTOP_MAX_ACTORS='0' # 0 或未设置表示全部演员
.\build-release-nmake\besktop.exe
```

10 个演员对照组：

```powershell
$env:BESKTOP_MAX_ACTORS='10'
.\build-release-nmake\besktop.exe
```

统计日志位于 `%LOCALAPPDATA%\Besktop\logs\besktop.log`。诊断日志可能包含桌面项名称或来源信息，只用于本机分析，不得提交到 Git。

采样时应记录：

- commit、架构和 Debug/Release 配置。
- 显示器分辨率、DPI、演员数量和任务栏位置。
- 觉醒阶段与稳定漫游阶段的 FPS、平均/最大绘制时间。
- 各渲染阶段平均耗时。
- 运行时间，以及 GDI 句柄和私有内存是否持续增长。

不同机器之间的绝对 FPS 不能直接判定回归。比较代码改动时，应尽量在同一机器、同一显示设置、同一桌面快照和同一构建配置下测量。

## 新增动作的性能回归要求

以后增加拳击、踢击、受击、关系互动或主题演出时，性能验收必须与视觉验收同时进行：

1. 保留未设置 `BESKTOP_MAX_ACTORS` 时全部图标觉醒的默认行为。
2. 同时测量全量演员和 10 个演员，区分固定全屏成本与单演员成本。
3. 覆盖睡眠、觉醒、四肢生长、稳定漫游和新增动作的高峰阶段，不能只测最轻的稳定帧。
4. 在同一参考环境下，稳定 FPS 不应比本基线下降超过 10%，平均绘制耗时不应上升超过 10%；超过时必须定位到具体阶段并说明取舍。
5. 全量演员应继续稳定达到至少 30 FPS。若改动未达到该底线，不应通过减少默认演员、放慢动画、删除文字或降低刷新目标掩盖问题。
6. 至少运行 5 分钟，确认 GDI 句柄和私有内存没有持续增长，并验证 `Esc`、`Ctrl+Shift+B` 正常退出。
7. 普通 Release 必须保持统计关闭、低噪声日志和无桌面项隐私明细。

新增视觉效果应先测量，再选择缓存、复用、减少状态切换或合并绘制调用等优化方式。只有数据证明当前 GDI/GDI+ 路径成为结构性限制时，才进入渲染后端迁移评估。

## 已完成的稳定性验证

第一轮优化已完成：

- x64 Debug、x64 Release、Win32 Release 的 GUI 与 Pack CLI 构建。
- 三份 Pack CLI 退出码为 0。
- x64 Debug `Esc` 和 Win32 Release `Ctrl+Shift+B` 安全退出。
- Win32 Besktop 对 x64 Explorer 的 47/47 Shell 来源与图像来源绑定，无回退。
- 全量演员持续运行 5 分 30 秒。
- GDI 句柄稳定为 60，私有内存稳定在约 30.6–31.1 MB。

仍需人工回归多种透明图标的 Bilinear 边缘、文字完整抖动与淡出、上/左/右任务栏，以及 Windows 7 32 位真机运行。
