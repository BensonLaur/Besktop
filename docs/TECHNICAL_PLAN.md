# Besktop 技术方案草案

本文档用于在实现前打磨技术路线。当前阶段以确认边界、降低风险、保留扩展性为主，不急于写完整功能。

## 技术目标

Besktop Core 需要满足几个关键目标：

- 体积小，方便用户下载、转发和尝试。
- 启动快，退出明确，行为透明。
- 原生 Windows 体验，不依赖浏览器壳或大型运行时。
- 对杀毒软件友好，不做隐藏驻留或静默自启动。
- 只在 Besktop 自己的全屏桌面舞台中演出，不破坏真实桌面文件。
- 免费核心体验完整，Plus 只扩展社交和个性化乐趣。
- 第一版发布物优先为单个 `Besktop.exe`，方便传播和试用。

## 桌面舞台与渲染

第一版采用全屏桌面重建舞台，而不是直接在真实桌面上叠加透明动画。

- 使用 Win32 创建全屏无边框窗口。
- 第一版只覆盖主显示器，并覆盖任务栏。
- 启动时读取桌面背景、图标位置、图标名称、图标图像和任务栏静态视觉。
- 在创建全屏窗口前，通过主显示器 `rcWork` 和主任务栏窗口矩形确定工作区域与任务栏位置；任务栏像素使用一次性屏幕区域捕获并由快照自有 BGRA 缓冲保存。
- 在 Besktop 窗口中重新绘制一份桌面舞台。
- 真实桌面始终留在下面，不修改、不移动、不隐藏真实桌面文件。
- `Esc` 立即退出，预留 `Ctrl + Shift + B` 强制退出热键。
- 使用 Direct3D 11 绘制图标贴图、粒子和轻量 3D 投影。
- 使用 Direct2D 绘制白色手脚、线条、简易 UI 和调试辅助。
- 使用 DirectWrite 绘制图标文字和提示文案。
- 使用 WIC 加载壁纸、图标和任务栏截图。

这样做的原因是：如果只使用透明覆盖层，真实图标仍留在原处，动画图标移动后会露出重影；如果直接使用整屏截图当背景，截图里也会包含原始图标。桌面舞台需要分离重建壁纸、图标、文字和任务栏，才能让图标演员真正离开原位。

任务栏复刻仅是启动时的静态视觉：不接管 Explorer、不响应点击，也不刷新时钟、托盘动画或按钮状态。任务栏检测支持主显示器的上、下、左、右边缘并裁剪到显示器范围；自动隐藏、捕获失败或异常矩形会安全跳过像素绘制，漫游范围仍优先使用有效 `rcWork`，获取失败时才使用集中式安全边距。`Shell_SecondaryTrayWnd`、第三方 Shell 和完整多显示器任务栏留待后续验证。

图标本体不是直接操作 Explorer 里的真实图标，而是采集图标外观后在舞台中绘制一个“演员”。这样才能做旋转、受击、双面薄片翻转和 3D 方向摆动。

## 图标演员身体模型

图标演员的身体不是可随意拉伸的 2D 卡片，而是一张有固定物理尺寸的双面薄片。

- 图标薄片使用本地 3D 坐标：
  - `x` 是图标横向。
  - `y` 是图标纵向。
  - `z` 是垂直于图标平面的深度方向。
- 角色根节点、肩部中心和胯部中心位于本地 `x = 0, z = 0` 的身体竖直中轴；该中轴是转身枢轴。
- 图标矩形位于 `z = 0` 平面，但图标几何中心相对身体中轴保留“半宽 + 肢体半径 + 可见净空”的水平偏移，不与转身枢轴重合，也不覆盖身体轴；稳定行走时图标位于移动方向后方。
- 正面和背面都绘制同一个真实图标贴图。
- 背面显示时不能左右镜像；左转、右转时只要不是侧面对屏幕，用户都应该能认出原图标。
- `rotateY` 接近 90 度时，图标在屏幕上自然压缩成一条窄线，像一张纸片立起来。
- 图标本体尺寸来自桌面图标显示尺寸或 DPI-aware fallback，不通过动作随意放大、压扁或拉伸。

四肢也不应当贴在图标投影后的 2D 边缘上，而应当挂在图标薄片之外的局部 3D 空间中：

- 肩膀和胯部有 `z` 深度偏移，不在 `z = 0` 图标平面内。
- 手臂为两段骨骼：肩膀、肘、手。
- 腿为两段骨骼：胯、膝、脚。
- 每段骨骼长度固定，动作通过关节角度变化实现，不通过拉伸线段实现。
- 所有关节先在本地 3D 中计算，再和图标薄片使用同一套 `rotateX`、`rotateY`、`rotateZ` 投影到屏幕。
- 当角色正对屏幕外时，图标应接近竖线，手脚因为深度偏移分布在竖线两侧。

当前视觉 MVP 可以先用 GDI+ 和手写 3D 投影近似实现；后续如果切换到 D3D11，也应保留这套“固定双面薄片 + 局部 3D 骨架”的模型。

## 图标采集策略

首选路径：

1. 使用 `IDesktopWallpaper` 获取当前显示器壁纸路径和显示方式。
2. 目标路径是通过 Shell View / `IFolderView` 读取 Explorer 桌面图标视图。
3. 获取可见图标的位置、名称和图标图像。
4. 由 Besktop 自己绘制图标文字，而不是截图文字。
5. 截取任务栏区域作为第一版静态视觉。
6. 为每个图标创建 `IconActor`。

当前视觉 MVP 的落地路径：

1. 壁纸仍通过 `IDesktopWallpaper` 获取。
2. 图标位置和名称先通过桌面 `SysListView32` 只读采样获取。
3. 图标图像优先通过 Shell 图像接口获取；失败时使用系统图标或内置占位图。
4. 采样失败时继续进入 demo actor 模式，保证动画可见。

容错路径：

1. 如果壁纸读取失败，使用纯色或内置演示背景。
2. 如果真实桌面图标位置读取失败，使用桌面快捷方式生成近似网格。
3. 如果图标图像读取失败，使用系统图标或内置占位图。
4. 如果桌面扫描完全失败，进入 demo actor 模式，保证用户仍能看到效果。

## 动作系统

第一版优先使用手写关键帧，而不是引入复杂动画引擎。

每个 `IconActor` 维护：

- `screenPosition`：舞台坐标。
- `iconTexture`：图标贴图。
- `bodyTransform`：位置、朝向、`rotateX`、`rotateY`、`rotateZ` 和投影参数。
- `bodyPlane`：固定尺寸的双面图标薄片。
- `limbPose`：本地 3D 两段式手脚骨架。
- 第一版 `state`：静止、醒来、长手脚、闲逛；寻敌和战斗状态延后。

当前视觉 MVP 可以用程序化公式和少量硬编码关键帧快速验证效果，但这不是长期动作方案。走路、拳击、侧踢、闪避如果都靠零散 `sin` 和角度公式拼接，会很快出现脚滑、根节点漂移、动作互相覆盖和“不像人”的问题。

长期动作系统应收敛到“动作片段 + IK + 事件点”的轻量模型：

- `AnimationClip`：描述一个动作片段，例如 `grow_limbs`、`walk`、`turn`、`punch`、`side_kick`、`dodge`、`hit_react`。
- `KeyPose`：记录身体薄片朝向、局部 3D 肩/胯锚点、手脚目标点或关节角度。
- `RootMotion`：描述演员整体移动，避免动作和位移各自为政。
- `ContactEvent`：记录脚落地、拳头命中、踢腿命中、受击触发等关键时刻。
- `IKTarget`：对手、脚使用目标点驱动，再用两段 IK 解出肘、膝位置。
- `Blend`：在静止、走路、转身、攻击、受击之间做短时间混合，避免姿态瞬间跳变。

阶段 A 已落地独立的 `action_clip.*` 与 `action_player.*`：动作标识、统一五阶段、内置只读 clip、姿态采样、每演员播放状态和一次性事件均不依赖窗口、桌面采集或 Explorer。第一轮 16 个公开单演员预览 ID 已补齐。`ActionClip` 集中保存攻击类型、轻重强度、防守窗口和打空标记；`ActionSample` 通过前后手/脚目标、前后肩局部三维偏移、身体与下半身旋转、前向/侧向/纵向 root offset 以及拳击/踢击/受击/打空强度表达姿态。肩部偏移先作用于肩带挂点，再以移动后的肩点作为固定骨长手臂 IK 根部；图标薄片与肩胯继续共享躯干变换，使直拳可以形成“前肩前送微抬、后肩后撤下沉、躯干前送下压”的连续动力链，而不是只移动拳端。场景每演员每帧只推进/采样一次动作，再把这些通用局部通道叠加到现有基础姿态；渲染层不按动作 ID 特判，未启用预览时采样通道保持为零。

走路尤其不能只做“腿相对胯部摆动”。更合理的步态模型是：

- 左脚落地时，左脚在世界坐标中短暂锁定，身体从落地脚上方经过。
- 右脚进入摆动相，抬脚、前摆、落地。
- 左腿前摆时右臂前摆，右腿前摆时左臂前摆。
- 身体有轻微上下起伏、前后倾和横向摆动，但图标本体仍保持固定尺寸薄片。
- 转身阶段由 `turn` clip 控制，`rotateY` 接近 90 度时图标自然变成窄线。

当前步态校准已把腿部目标和两段 IK 抽到独立的纯数学模块 `gait_ik.*`。步相按演员实际移动距离推进，`locomotionWeight` 在移动与等待之间短时混合；等待、动作预览和四肢生长阶段使用中性站姿，不再继续播放走路循环。腿部保持 `0.40 / 0.41 planeSide` 的两段固定骨长，并明确按约 `62%` 支撑相和 `38%` 摆动相采样：支撑脚沿地面后移，摆动脚在离地和前摆时屈膝抬起，落地前再逐渐伸展，不使用统一下垂量把两条腿全程拉直。步幅约为 `0.68 planeSide`，摆动相最大抬脚高度约为 `0.14 planeSide`，中性有效下垂距离为总腿长的 `0.994`。48 像素基准几何测试中，中性站姿膝角约 `169.6°`，摆动初段、中段和落地前膝角分别约 `131.5°`、`110.9°` 和 `171.1°`，关键采样范围约 `110.9°–171.1°`，两条大腿最大夹角约 `44.4°`；这些数值是局部 3D 数学结果，最终仍以投影后的视觉检查为准。

连续转身由独立纯数学模块 `turn_motion.*` 管理，不作为攻击 `ActionClip`，也不产生 Contact。每演员分别保存已提交的 `currentFacing`、本轮锁定的 `desiredFacing`、转身状态、起止 yaw、持续时间和单调进度；反向目标先停止根位移并把 `locomotionWeight` 平滑降到 `0`，再用 `smoothstep` 在 `0°` 与 `180°` 间旋转 `0.40` 秒，完成后才提交朝向并恢复步态。同向目标不会触发多余转身，转身期间也不会因 `dx` 接近零而反复改目标。

向右稳定姿态作为统一局部骨架基准，向左由相对基准的 `180°` Y 轴旋转得到。身体中轴的屏幕位置在转身期间固定；图标中心作为位于移动方向后方的偏移子节点绕该轴走连续圆弧，在 `90°` 附近经过中轴，完成后自然落到另一侧，而不是以图标中心为轴原地翻卡片。`BuildTurnActorGeometry` 统一提供肢体线宽、可见净空、轴间隙、图标中心偏移和最大水平范围；初始化、身体投影、边缘朝向判断、漫游边界和测试不再维护重复公式。图标四角、肩胯挂点和四肢关节都经过同一身体中轴旋转及透视投影，不再在 `facing` 改变时直接翻转挂点 X 或前后深度。图标正反面继续绘制同一纹理；越过 `90°` 后只调整目标三角形顶点顺序以保持非镜像 UV，不翻转文件图标、文字或 Logo。肢体前后层依据投影后整条肢体的平均深度决定，深度关系在转身中段自然交换。演员初始化时保持图标真实捕获中心不变，并根据左右可用空间选择安全初始朝向，使靠边演员的后续圆弧优先向屏幕内侧展开。

拳击、侧踢和闪避也应复用同一套骨架与 IK：

- 拳击使用手部目标点和命中事件，而不是单独拉长手臂。
- 侧踢使用通用双脚目标通道和固定骨长两段 IK：踢击脚按“收膝—伸展—收膝—落地”采样，支撑脚保持中性局部脚底目标；身体反倾和转胯不直接旋转脚端，而是重新求解膝盖位置。
- 受击使用短 clip 叠加身体薄片晃动、后退和恢复。
- 命中检测可以先用简化 hitbox，后续再精细化。

外部动作数据可以作为参考来源，但第一版不建议直接接入大型 mocap 或完整 3D 角色动画链。Besktop 的角色是“图标薄片 + 白色线条四肢”，更适合把参考动作转写为紧凑的自定义 clip 数据，保证体积小、可控、容易内置到单 exe。

动作数据未来可以从 JSON 或紧凑二进制资源加载，方便免费包和 Plus 包共用同一套运行时。免费核心动作应先覆盖走路、转身、拳击、侧踢、闪避、受击；Plus 后续只扩展更多动作风格和社交玩法，不改变 Core 的骨架与动作格式。

第一轮格斗动作的统一命名、候选优先级和冻结范围见 [FIGHT_ACTION_CATALOG.md](FIGHT_ACTION_CATALOG.md)；与当前 `IconFightScene` 对接的数据结构、接触事件、简化命中和分阶段实施顺序见 [FIGHT_ACTION_IMPLEMENTATION.md](FIGHT_ACTION_IMPLEMENTATION.md)。

第一版推荐把基础动作数据作为内置资源嵌入 `Besktop.exe`。Plus 内容如果随商业版一起发布，也优先在 Besktop-Plus 的打包阶段压缩、签名后嵌入最终 exe，而不是要求用户手动管理外部包文件。

动作打磨阶段保留时间缩放和时间偏移能力。默认动画速度为 `1.0x`，开发者可以用 `BESKTOP_ANIMATION_SPEED=0.5` 之类的环境变量慢放观察步态、转身、拳击和受击细节；也可以用 `BESKTOP_ANIMATION_OFFSET=4.5` 直接从指定动画秒数附近开始，减少反复等待。该能力只影响 Besktop 舞台内的动画时间，不改变系统时间、真实桌面或渲染帧率。

## 诊断与性能观测

视觉 MVP 阶段需要保留可关闭的性能诊断能力：

- `BESKTOP_FRAME_STATS=1`：每秒记录 `paint fps`、`timer fps`、`timer avg/max ms`、`paint avg/max ms`，并细分缓冲准备、静态背景复制、壁纸、任务栏、演员姿态/投影准备、四肢、图标主体、文字和最终 BitBlt，用于确认实际帧率与热点；Release 仍须先启用诊断总开关。
- `BESKTOP_FRAME_TRACE=1`：记录首帧绘制、壁纸缓存和关键渲染阶段，用于定位首帧卡顿或异常。
- `BESKTOP_ANIMATION_SPEED=<倍率>`：控制动画时间倍率，默认 `1.0`，建议允许慢放观察动作细节。
- `BESKTOP_ANIMATION_OFFSET=<秒>`：设置动画起始偏移，默认 `0.0`，用于直接观察左走、右走、转身、拳击等指定阶段。
- `BESKTOP_MAX_ACTORS=<数量>`：限制本次创建的演员数量，用于比较不同桌面图标规模下的渲染性能；未设置或设为 `0` 时创建全部演员。
- `BESKTOP_ACTION_PREVIEW=<动作 ID>`：让首个有效演员在觉醒后原地循环预览第一轮 16 个公开动作 ID；完整清单见 `README.md` 与 `FIGHT_ACTION_IMPLEMENTATION.md`。
- `BESKTOP_COMBAT_PREVIEW=<场景 ID>`：让前两个有效演员运行固定双人攻防闭环；当前支持 `lead_parry`、`lead_slip`、`uppercut_light_hit`、`side_kick_heavy_hit`。它优先于转身和单动作预览，并且同样只在 Debug 或 Release diagnostics 总开关下生效。
- `BESKTOP_COMBAT_DIRECTOR_PREVIEW=1`：显式启用轻量导演的诊断观测；同时最多预约并控制一对已经觉醒的演员，完成一轮固定攻防、分开和冷却后交还漫游。普通 Release 未开启 diagnostics 时忽略该变量，但产品 Director 本身默认开启。
- `BESKTOP_ACTION_ORBIT_CAMERA=1`：仅在动作预览中增加独立观察 yaw，摄像头以 8 个动画秒一圈绕身体中轴连续观察；图标薄片、肩胯和四肢共同经过观察投影，但不修改动作 clip、逻辑前后手、`currentFacing` 或 `TurnMotionState`。
- `BESKTOP_TURN_PREVIEW=1`：让首个有效演员原地循环“向右站立—转向左—停顿—转向右—停顿”，用于慢放检查 90° 薄片、挂点连续性和深度交换。

Debug 构建通过 `BESKTOP_DEVELOPER_BUILD` 明确标记开发者能力，默认允许读取各项诊断环境变量并记录详细 Info 日志。普通 Release 固定使用 `1.0x` 动画速度和 `0` 秒偏移，默认关闭帧率统计、分段 trace 和图标平面调试，并只记录 Warning/Error；过滤掉的 Info 不会触发日志文件创建。

Release 现场诊断必须先设置 `BESKTOP_ENABLE_DIAGNOSTICS=1`，之后才读取其他诊断变量并启用详细 Info 日志。单独残留 `BESKTOP_FRAME_TRACE`、`BESKTOP_DEBUG_ICON_PLANE` 或动画倍率变量不会改变普通 Release 行为。所有运行选项由进程启动时一次性加载的只读 `RuntimeOptions` 提供，场景、舞台和壁纸渲染器不再分别解析环境变量。

已验证的优化前后数据、分阶段热点、复现方法和新增动作的性能回归要求见 [RENDER_PERFORMANCE.md](RENDER_PERFORMANCE.md)。

第一轮单演员动作库补齐仅执行 x64 Release 精简验证：全量演员 27 秒短测的大多数稳定窗口为 `31.0–32.0 FPS`，未观察到持续跌出当前约 30 FPS 基线。完整 x64/Win32、长期资源与兼容性矩阵留到固定双演员阶段收口或正式发布候选统一执行。

互动系统按职责分层：`ActionClip/ActionPlayer` 负责冻结的单演员动作数据与事件；`CombatPair` 只负责真正攻防时的一对演员共享时间线、Contact 和结果状态；`EncounterDirector` 负责一对演员的相遇前奏、结果分支、余波与完成信号；`ActorBehaviorProfile/ActorRuntimeState` 保存稳定倾向和仅本次演出有效的警觉、兴奋、体力、最近对象与个体冷却；`ActorPerception/LocalIntent` 从有限邻域生成短窗口个体意图和双向握手请求；`EncounterArbiter` 只裁决演员占用、工作区与 reservation 冲突。`IconFightScene` 作为舞台适配层提供演员状态、工作区边界、移动转向及姿态几何，不复制第二套攻防状态机。固定 `BESKTOP_COMBAT_PREVIEW` 仍直接进入原 `CombatPair`，不受产品相遇分支影响。Contact 在精确事件时刻复用同一套姿态、投影和固定骨长骨架求解，生成拳端/脚端与目标身体轴胶囊，再交给纯数学结果判定；渲染缓存不是唯一判定来源。

普通 Release 的觉醒计划由纯逻辑 `AwakeningDirector` 在初始化时生成一次，不读取环境变量、桌面名称或路径。第一批约占 `20%`，使用归一化工作区上的确定性最远点采样分散到不同区域，并在约 `2–9` 秒缓慢启动；第二批约占 `35%`，在约 `12–26` 秒启动；其余演员在约 `30–52` 秒兜底启动，计划批次之间保留明确停顿。批次内时间、邻近加速延迟和同帧去同步均由演员现有随机状态派生的独立稳定种子决定，不消耗漫游随机序列。邻近检查以 `0.15` 秒间隔运行，只让已完成四肢生长并进入正常探索的演员，在约 `2.7` 个图标边长内把沉睡演员的时间提前到“当前时刻加 `4–7` 秒”；该因果等待用于保留扩散层次，它不能延后计划、不能突破 `1.0` 秒桌面停顿，也不是最终觉醒的必要条件。标签震动使用各演员自己的 `actorTime`，未到计划时间的演员保持原位静止，不再共享全局震动窗口。

普通 Release 默认启用产品 Director。它在第一批全部完成四肢生长且当前至少有两名正常 Wanderer 后，才开始 `6.0` 秒纯探索缓冲，不再等待所有演员；尚未觉醒、正在觉醒或长四肢的演员仍会被候选过滤，第二批和兜底批次继续独立推进。后续只有在全局冷却结束时才执行一次候选评分，不在每帧运行全量选对。候选必须距离适中、处于正常漫游、没有转身或动作；评分综合自然距离、历史参与次数、最近四对配对、边缘空间和稳定随机扰动，避免长期只选择中央少数演员或刚结束的同一对。完整圆形 reservation 必须落在任务栏之外的工作区内，空间不足时以 `0.85–1.65` 秒稳定随机间隔安静重试，不强制拉人跨场。演员不足、工作区过小或暂时没有候选时均安静退化为纯漫游。

reservation 激活后，非参与演员不仅在生成新目标时避开它，也会检查旧目标和当前位置到目标的线段。目标落入圆内、路径穿圆或演员在软边界内继续向内移动时，会在工作区安全范围内选择带径向退让的切向绕行点；`0.85` 秒重选间隔和约 `0.28 reservation 半径` 的滞回区用于避免逐帧抖动。避让只改变普通漫游目标，仍由既有连续转身、步态和 delta time 移动完成，不瞬移演员。

参与双方按 `Notice → Approach/SlowDown → FaceEachOther → Assess → Intent → Combat/Yield/Bluff → Aftermath → Separate` 执行。先到者在站位平滑降低步态权重等待；双方完成连续转身后，以稳定随机值观察 `0.6–1.0` 秒，并由主倾向只做约 `-0.08–+0.08` 秒的小幅等待修正。稳定主倾向只来自演员种子，不读取应用语义或文件内容；临时状态按 delta time 衰减或恢复，不持久化到磁盘。局部感知半径约为较大演员边长的 `6.25` 倍，并同时检查距离变化、视野、觉醒/漫游/转身/动作/恢复/占用/个体冷却和完整 reservation 是否处于工作区。个体意图在 `0.35–0.8` 秒窗口内保持，倾向和临时状态只改变概率权重：Bold 更容易 Challenge/Respond，Timid 更容易 Avoid/Yield，Curious 更容易 Observe/Approach，Calm 更容易 Ignore/Observe，Energetic 更容易 Approach 而不等同于好斗。

只有双方互相锁定且意图兼容时才形成请求；Challenge + Avoid 明确不建立正式相遇。`EncounterArbiter` 对请求作确定性排序，一个演员只能被一个已接受请求占用，越界、与现有 reservation 重叠或被其他控制器占用的请求会被拒绝，演员继续漫游。仲裁器没有 `maxActivePairs` 一类产品级上限；`ActiveEncounterPool` 会接收同一轮全部互不冲突结果，为每场保存稳定 ID、双方索引、独立 `EncounterDirector/CombatPair`、站位和 reservation。池只持有索引，不长期保存演员引用；遍历期间先标记释放，迭代后统一清理，单场完成或异常取消不会破坏其他场次。下一阶段才把 Combat 扩展为约 `8–16` 秒、`3–7` 次交换的完整交锋段。

Yield 和 Bluff 继续只使用独立 encounter 加法姿态与小范围移动，不创建 ActionId、Contact 或特效。Combat 结果不再共用固定 `0.45` 秒和对称离场：轻受击者先走，重受击恢复更慢且离场更远，拨挡双方短暂僵持，闪避者从侧向优先离开，打空者完成既有恢复后主动转身；HeavyStagger 已提交的 RootMotion 不会再次叠加。所有离场点在工作区及同一 reservation 内计算，异常时只取消并释放本场。普通桌面继续使用 `14` 秒个人冷却和最近对象记录抑制反复配对；并行相遇不使用全局单场冷却作为并发闸门。按 `P` 后不再产生新握手请求，已经开始的各场相遇自然完成，重新启用仍保留 `4` 秒漫游缓冲。固定 Combat、Turn、Action 诊断预览优先于产品路径。

多对互动由 `ActiveEncounterPool` 保存已通过仲裁的全部活动 reservation，不设置产品级固定对数上限，也不改变 ActionClip 和 CombatPair；实际并发量只受演员占用、空间安全和可用局部请求约束。非参与演员的旧目标与新目标都会针对全部 reservation 做轻量避让。按 `P` 关闭会立即停止建场，池内项目各自自然收尾；重新开启保留既有恢复漫游等待。固定诊断预览优先并暂停产品池。

首版产品模式接入后的 x64 Release 快速对照中，全量演员各采样约 `22` 秒：Director 开启的稳定样本平均 `27.78 FPS`（`26.6–28.6`），按 `P` 切换纯漫游后平均 `27.15 FPS`（`25.6–28.0`）。两组均处于当前机器约 `25–27 FPS` 的短测基线附近，未观察到 Director 带来的明确额外回退；该数据只用于快速回归，不替代发布候选阶段的长期性能与资源稳定性测试。

## Plus 扩展边界

Core 可以知道“玩法包”和“授权状态”的概念，但不能包含支付密钥、私钥、客户数据或私有资源。

建议边界：

- Core 负责加载免费包和验证签名包。
- Plus 负责生成、签名、嵌入或分发付费内容。
- Core 内置公钥，Plus 服务端持有私钥。
- 未授权时，Core 仍然提供完整基础玩法。

## 发布形态

优先发布形态：

```text
Besktop.exe
  Core 引擎
  免费 Icon Fight 内容
  Plus 内容容器
  授权验证客户端
```

用户下载和传播的是一个 exe。内部可以有包、插件、资源段或追加数据，但这些属于实现细节。

公开 Core 首版使用 CMake `project()` 的 `0.1.0` 作为唯一版本来源，由配置阶段生成 C++ 版本头和 GUI `VERSIONINFO`；Pack RCDATA 保持为 GUI 与验证 CLI 共用的公共资源，GUI 文件元数据使用独立资源模板。MSVC 构建统一使用静态 C/C++ 运行库（Release `/MT`、Debug `/MTd`），Windows 系统 DLL 仍由系统提供。

`tools/build-release.ps1` 从专用 Release 目录重建 GUI 和 Pack CLI，验证内置 Pack、版本字段和运行库依赖，再输出 `dist/Besktop.exe` 及 SHA-256。当前工程不包含代码签名、安装器、自动更新或最终品牌图标，这些均不属于“单 EXE 可运行”的含义。

可选的后续形态：

- 单 exe 内置所有首批 Plus 内容，适合第一版传播。
- exe 加在线下载签名 Plus 包，适合后续内容频繁更新。
- exe 加外部 `packs/` 目录，仅在用户确实需要管理大量内容时考虑。
