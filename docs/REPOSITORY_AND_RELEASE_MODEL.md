# Besktop 仓库定位与发布模型

本文档记录 Besktop 与 Besktop-Plus 的协作方式，以及公开仓库应该暴露哪些内容。

## 结论

- 对用户发布一个软件：`Besktop`。
- 第一版优先发布一个文件：`Besktop.exe`。
- `Besktop` 是公开主仓库，也是产品发布主体。
- `Besktop-Plus` 是私有商业仓库，负责 Plus 内容、支付授权和发布编排。
- Plus 在工程上可以是插件或玩法包，但第一版不要求用户看到外部包文件。

## 为什么不是两个软件

不建议发布 `Besktop Free.exe` 和 `Besktop Plus.exe`。

原因：

- 用户会困惑该下载哪个。
- 传播路径会变长。
- 免费体验和付费体验割裂。
- 售后需要解释两个程序之间的关系。

更好的产品表达是：

```text
Besktop.exe
  免费功能直接可用。
  Plus 功能购买后在同一个软件中解锁。
```

## 工程关系

```text
Besktop
  公开仓库。
  维护 Core 引擎、免费玩法、公开接口和文档。
  产出可独立运行的免费版 Besktop.exe。

Besktop-Plus
  私有仓库。
  维护付费内容、支付授权、私有资源和商业发布脚本。
  发布时同步指定版本 Besktop，嵌入或启用 Plus 内容，产出最终 Besktop.exe。
```

## 单 exe 与内部包

“插件”或“玩法包”是工程概念，不一定是用户可见文件。

第一版推荐：

```text
Besktop.exe
  Core 引擎
  免费基础内容
  Plus 内容容器
  授权验证逻辑
```

未来可以扩展为：

- 内置包：资源压缩后嵌入 exe。
- 追加包：包数据追加到 exe 末尾，由程序自定义读取。
- 外部包：`packs/*.bpack`，仅在内容频繁更新或体积明显变大时考虑。

## 公开仓库可以暴露

公开仓库可以包含：

- 产品定位、路线图和公开技术方案。
- 主程序代码。
- 桌面舞台、渲染、桌面图标扫描和动作系统。
- 免费 Icon Fight 内容。
- 内部玩法包格式的公开定义。
- 签名验证、公钥验证和授权状态接口。
- mock 授权、示例包和测试数据。
- 打赏入口。

这些内容有助于建立用户信任，也方便未来维护。

## 公开仓库不应暴露

公开仓库不应包含：

- 支付平台密钥和商户配置。
- 授权私钥。
- 生产 Worker 的敏感配置。
- 客户订单、邮箱和售后数据。
- Plus 付费内容源码、私有动作脚本和素材。
- 商业定价实验、收入统计和运营细节。

## 私有仓库应该包含

Besktop-Plus 应该包含：

- 付费玩法内容。
- Plus 内容签名和打包流程。
- 支付回调和授权签发服务。
- 购买页和商业发布脚本。
- 售后查询、授权恢复和设备重置流程。
- 与具体支付平台相关的私有适配代码和配置模板。

## 发布流程草案

```text
1. Besktop 公开仓库打 tag。
2. Besktop-Plus 固定该 tag。
3. Besktop-Plus 构建 Core。
4. Besktop-Plus 生成并签名 Plus 内容。
5. 打包脚本将 Plus 内容嵌入最终 Besktop.exe。
6. 输出免费公开版或 Plus 可解锁版。
```

第一版可以先只发布一个“Plus 可解锁版” `Besktop.exe`：免费功能立即可用，Plus 内容内置但需要授权解锁。
