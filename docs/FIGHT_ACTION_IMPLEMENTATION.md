# Icon Fight 第一轮动作系统实施计划

本文档把 [FIGHT_ACTION_CATALOG.md](FIGHT_ACTION_CATALOG.md) 冻结的第一轮动作范围映射到当前 Besktop Core。它是工程实施计划，不代表所有内容必须在一个提交或一个 session 中完成。

## 当前基础

当前 `IconFightScene` 已具备：

- 全量桌面图标演员、稳定随机种子、错峰觉醒和自由漫游。
- 固定尺寸双面图标薄片和局部 3D 两段式四肢。
- 步态、转向、身体投影、两段腿部 IK 和前后层四肢绘制。
- `ActorPose` 中预留的 `punch`、`kick`、`dodge`、`hit` 表现参数。
- 每演员每帧一次姿态、身体投影和骨架准备缓存。
- 分阶段帧耗时统计和 `BESKTOP_MAX_ACTORS` 对照测试。

阶段 A 已建立独立 `ActionClip` / `ActionPlayer`、每演员动作状态、统一阶段和单人诊断预览。第一轮 16 个公开预览 ID 的内置动作数据、通用姿态通道、防守窗口、Contact 元数据和数学测试现已补齐。攻击者与防守者配对、真实接触判定、攻击结果和自动互动仍未实现，属于后续阶段。

## 实施目标

第一轮完成后，系统应支持：

- 演员从漫游自然进入准备、攻击、防守、受击、打空回收，再回到漫游。
- 同一套动作数据可镜像到左右朝向。
- 攻击者和目标共享同一条交互时间线，接触时刻一致。
- 直拳、勾拳、踢击使用固定骨长和 IK 目标，不拉伸手脚。
- 转身后踢驱动图标薄片、肩胯和踢腿共同旋转。
- 防守窗口可以产生格挡、闪避或打空，而不是每次攻击必定命中。
- 受击反馈根据轻击/重击区分偏转和后退。
- 默认演出逐步出现少量配对互动，不让全部演员同时攻击。

## 非目标

- 本轮不做真实格斗游戏级物理、逐像素碰撞、生命值和胜负系统。
- 不做倒地、起身、抱摔、地面动作或多人围攻。
- 不在本轮把动作数据开放为用户可编辑外部文件。
- 不切换 D2D/D3D 渲染后端；动作系统先复用当前 GDI+ 渲染路径。
- 不加入 Plus 私有玩法、商业动作包或授权判断。

## 建议数据模型

### 动作标识与阶段

```cpp
enum class ActionId {
    None,
    SwingPunch,
    LeadStraight,
    RearStraight,
    Hook,
    Uppercut,
    FrontKick,
    SideKick,
    RoundhouseKick,
    SpinningBackKick,
    Layback,
    Slip,
    Parry,
    LightHitReact,
    HeavyStagger,
    WhiffRecovery,
};

enum class ActionPhase {
    Prepare,
    Active,
    Contact,
    Recover,
    Complete,
};
```

左右侧闪使用相同时间结构和镜像采样规则，公开动作 ID 与调试名称分别保留 `slip_left`、`slip_right`，从而保证名称可以稳定反向查询。

### 动作片段

```text
ActionClip
  id
  duration
  prepareEnd
  activeStart / activeEnd
  recoverStart
  body key poses
  hand / foot IK targets
  root motion
  contact events
  block / evade windows
```

第一轮先使用 C++ 内置只读表，验证动作语言和数据结构。等动作稳定后再迁移为免费 Pack 内嵌数据，避免在姿态仍频繁调整时同时设计序列化格式。

### 每演员运行状态

```text
ActorActionState
  actionId
  phase
  localTime
  playbackRate
  direction
  targetActor
  actionSeed
  cooldown
  rootMotionApplied
  contactConsumed
```

演员的漫游数据继续保留。动作开始时暂停选择新漫游目标；动作完成后恢复原有目标或重新选择安全目标。

### 交互配对

```text
CombatPair
  attacker
  defender
  desiredDistance
  attackAction
  defenseAction
  expectedResult
  startTime
  contactResolved
```

`CombatDirector` 只负责选择少量双方、预约空间和动作结果，不直接绘制姿态。第一版同时活跃的交互对数量应受控，避免全量演员一起聚集和产生性能峰值。

## 姿态合成顺序

每帧姿态按固定顺序生成：

```text
觉醒/身体基础姿态
  + 漫游步态与朝向
  + 当前 ActionClip 局部姿态
  + RootMotion
  + 受击覆盖层
  → 身体投影
  → 四肢 IK
  → 绘制
```

攻击 clip 只覆盖需要控制的身体角度、手脚目标和根运动。未覆盖的肢体继续使用站姿或平衡姿态，避免进入动作时所有关节瞬间重置。

受击优先级高于普通攻击；安全退出、边界修正和任务栏避让优先级高于 root motion。

## 动作统一时间结构

所有第一轮动作使用相同语义阶段：

1. `Prepare`：蓄力、重心调整或抬膝，给观众可读预兆。
2. `Active`：拳脚快速接近目标。
3. `Contact`：触发一次接触判定和命中事件。
4. `Recover`：收拳、收腿、恢复平衡；打空时可以替换为更长的 `WhiffRecovery`。
5. `Complete`：短混合回站姿，再回漫游或进入反击动作。

动作速度可以不同，但阶段含义不能变化。接触事件必须只消费一次，低帧率下即使跨过精确时刻也不能漏掉。

## 简化命中与防守模型

第一轮使用可解释的几何判定：

- 攻击开始时锁定目标和朝向，不在接触帧瞬移追踪目标。
- 每个攻击事件记录攻击肢体、目标高度、基础距离和命中半径。
- 接触时根据攻击者手/脚世界位置与目标身体范围判断是否在距离内。
- 目标处于 `Parry` 有效窗口且攻击是拳法时，结果为 `Blocked`。
- 目标处于 `Layback` 或 `Slip` 有效窗口且身体离开攻击线时，结果为 `Evaded`。
- 不在范围、方向错误或被闪避时，攻击者进入 `WhiffRecovery`。
- 普通命中触发 `LightHitReact`；转身后踢、侧踢等重动作可触发 `HeavyStagger`。

第一轮结果枚举建议为：

```text
HitLight
HitHeavy
Blocked
Evaded
Whiffed
```

先由导演预选一部分可读结果，再由接触时的几何条件最终确认，避免为了固定剧本让明显打空的动作仍触发命中。

## 第一轮分阶段实施

### 阶段 A：动作播放骨架与第一轮单人预览库

状态：**已完成**。

目标：建立数据结构，不立即开启自动打架。

- 新增 `ActionId`、`ActionClip`、`ActionPlayer` 和 `ActorActionState`。
- 将动作时间推进与 `IconFightScene::Update` 的 delta time 对齐。
- 增加受诊断总开关保护的动作预览选项，例如：

```powershell
$env:BESKTOP_ACTION_PREVIEW='lead_straight'
$env:BESKTOP_MAX_ACTORS='1'
```

- 预览模式循环“站姿 → 动作 → 恢复”，便于慢放、截图和性能测量。
- 首先实现 `lead_straight`、`layback`、`light_hit_react` 三个代表动作，验证攻击、防守和受击三条路径。

验收：第一轮动作能独立循环、左右镜像、固定骨长并恢复中性通道；普通 Release 不读取预览变量。

已完成人工打磨的拳法使用以下动作轮廓作为视觉验收基准：

- **前手直拳 `lead_straight`**：前手从稳定防守位快速直线伸出；前肩前送微抬、后肩小幅后撤下沉，图标躯干仅做轻量前压；另一只手始终护在身前，Contact 后沿原直线路径快速收回，不出现额外肘击轮廓。
- **后手直拳 `rear_straight`**：后手先随肩带向后加载，再由更明显的躯干转动带动后肩穿过身体轴；后肩前送微抬、前肩后撤下沉，图标躯干的前送和下压强于前手直拳；前手保持防护，回收时先收拳再恢复肩带。
- **上勾拳 `uppercut`**：身体和出拳侧先轻微下沉收拳，随后保持弯肘由下向前上方勾起；出拳肩随躯干向上释放，另一只手继续防护；拳峰不能伸直成斜向直拳，Recover 也不能折回尖锐锐角。
- **前手平勾拳 `hook`**：前手保持接近直角的弯肘，在肩部高度沿短横向弧线扫过；前肩和图标躯干只做紧凑转动，后手护在身前；回收沿短弧返回，轮廓必须明显区别于直拳和夸张大摆拳。
- **夸张大摆拳 `swing_punch`**：前手与前肩先向身体外后方拉开，随后由更大的肩带和躯干转动带出宽横向弧线；拳峰保持中等弯肘，弧线、肩部推进和图标前压均明显大于平勾拳；后手仍保持防护，Recover 沿弧线收回而不是从远端穿回身体。
- **上身后仰闪避 `layback`**：双脚尽量保持位置，双手先收紧防护，髋部随一次浅蹲小幅下降、双膝统一朝角色前方弯曲，肩部中心与图标薄片再向后倾离攻击线；下半身不随上身大幅旋转，最大后仰仍保持身体控制；Recover 先带回肩背，再逐渐伸膝恢复中性站姿，不能表现成反关节、受击、倒地或整体后退。

只有九个攻击动作产生一次 `ContactEvent`：五种拳法与 `front_kick`、`side_kick`、`roundhouse_kick`、`spinning_back_kick`。`layback`、`slip_left/right`、`parry`、`light_hit_react`、`heavy_stagger`、`whiff_recovery` 虽然仍使用统一五阶段时间结构，但不发出攻击 Contact。

`ActionClip` 现在还提供攻击类型、轻/重命中强度、`EvadeWindow`、`ParryWindow` 和打空回收标记。窗口使用半开区间 `[start, end)` 查询：`layback` 为 `0.16–0.40` 秒，左右侧闪为 `0.14–0.36` 秒，拨挡为 `0.12–0.30` 秒。它们只为下一阶段的双方结果判定提供只读语义，本阶段不计算真实格挡、闪避或命中。

`ActionPlayer` 以 delta time 推进，单次更新可以跨越多个阶段；事件使用每次播放独立的位掩码记录，低帧率或 8 倍速跨过 Contact 时仍只产生一次。动作姿态使用局部通道叠加到现有觉醒/步态姿态，手臂目标通过固定骨长两段 IK 解算，进入和退出时按权重混合。

阶段 A 之后完成了一轮独立的步态腿型校准，但没有进入阶段 B：新增 `gait_ik.*` 纯数学模块，步相改为按移动距离推进，并用 `locomotionWeight` 区分移动、等待和动作预览。每条腿明确区分约 `62%` 支撑相和 `38%` 摆动相：支撑腿大部分时间接近伸直，摆动腿在离地和初始前摆时明显屈膝，接近落地时再伸展；两腿使用半周期偏移，不再长期同时屈膝或让两条大腿朝向同一侧。动作预览默认回到中性支撑腿，`layback` 的上身旋转只向下肢传递少量跟随。48 像素基准测试的中性膝角约 `169.6°`，摆动初段、中段和落地前膝角分别约 `131.5°`、`110.9°` 和 `171.1°`，关键采样范围约 `110.9°–171.1°`，大腿最大夹角约 `44.4°`，左右朝向的骨长、膝角和脚目标镜像均由纯逻辑测试覆盖。

`side_kick` 通过 `ActionSample` 的通用双脚目标通道驱动，不在绘制函数中按动作 ID 写特例。踢击脚目标以中性脚底为基准，先收膝、再伸展、先收回到收膝位置、最后落地；支撑脚目标在整个动作中保持为动作开始时的中性位置。身体最多约反倾 `8°`、转胯 `10°`，下半身动作旋转权重在该 clip 中设为 `0`，由固定骨长 IK 在移动的髋部与固定脚端之间重新求膝，避免脚端随图标薄片刚性旋转。48 像素基准测试中，Prepare 踢击膝角约 `104.4°`，Contact 约 `165.9°`，大腿和小腿分别恒定为 `19.2` 和 `19.68` 像素，支撑脚局部解算最大漂移为 `0`。Contact 仅发出一次预留事件；当前不做目标配对、命中判定、伤害或自动战斗，也不显示命中特效。

其余第一轮动作继续使用同一组通用通道：前后手/脚独立 IK 目标、前后肩局部三维偏移、身体与下半身旋转、前向/侧向/纵向 root offset、拳击/踢击/受击/打空强度。渲染层只消费这些通道，不读取 `Hook`、`RoundhouseKick` 等动作 ID。48 像素纯数学测试中，`hook` Contact 肘角约 `81.6°`、深度弧约为 `0.38 planeSide`；`swing_punch` Contact 肘角约 `99.7°`、深度弧约为 `0.62 planeSide`，肩部和图标前送也明显大于紧凑平勾拳；`front_kick` 膝角从约 `103.8°` 收膝伸展到约 `171.1°`。新增三种踢击的大腿/小腿继续固定为 `19.2 / 19.68` 像素，支撑脚局部目标漂移为 `0`；计入回旋踢有限下半身旋转后的近似投影漂移最大约 `1.26` 像素。`spinning_back_kick` 的动作内部 yaw 连续经过约 `90°` 并完成 `360°`，结束采样回到 `0°` 等价中性姿态，不写入或改变演员的外部 `TurnMotionState` 与 `currentFacing`。

漫游转身使用独立 `turn_motion.*`，不占用五阶段攻击动作，也不改变 `ActionPlayer` 的 Contact 语义。旧实现每帧按 `dx` 直接覆盖 `facing`，肩胯挂点又用该离散符号同时翻转 X 和前后深度，因此会出现四肢单帧换边；现在 `currentFacing` 只表示已完成朝向，`desiredFacing` 在转身开始时锁定，`0.40` 秒 `smoothstep` yaw 完成后才提交。转身前先把步态权重混合到中性站姿，转身期间根位置不动，结束后再恢复同一距离驱动步相。

角色根节点、肩部中心和胯部中心定义身体竖直中轴。图标与身体轴的几何由 `BuildTurnActorGeometry` 集中计算：`bodyAxisGap = limbWidth / 2 + visibleMargin`，避免中心半径存在但抗锯齿后仍与肢体粘连。48 像素基准下，肢体线宽为 `5 px`、净空目标为 `4 px`、轴到图标近边距离为 `6.5 px`，带稳定步态倾斜的屏幕投影净空约为左右各 `3.97 px`。稳定行走时图标挂在移动方向后方，向右走位于身体轴左侧、向左走位于身体轴右侧。

图标平面、肩胯和四肢关节共同绕身体轴旋转；右向基准 yaw 为 `0°`，左向为 `180°`。图标中心从轴一侧沿连续圆弧移动到另一侧，`90°` 时经过身体轴附近，同时原始 48 像素平面投影宽度接近 `0`，绘制层保留 `2` 像素最小边缘。背面继续使用非镜像 UV 顶点顺序。前后肢体按投影后根、关节和末端的平均深度分层，而不是等到朝向提交时离散交换。初始图标中心仍保持真实捕获位置；若演员靠近左右边缘，则选择让完整转身圆弧朝屏幕内侧展开的初始朝向，漫游边界也使用同一几何的最大水平范围，避免偏移增大后越界。

本轮视觉快速微调在 Release x64、全量演员下短测 25 秒，稳定采样约 `31.5–32.5 FPS`，平均绘制约 `30.9–31.7 ms`，姿态阶段约 `0.01 ms`；句柄由 `399` 变为 `398`，私有内存约 `30.8–31.0 MB`，未观察到快速增长。该数据仅用于确认没有明显跌出当前约 30 FPS 基线，不替代阶段收口时的完整兼容性和长期资源验证。

同一参考环境的 Release x64 全量 47 演员首轮复测稳定约 `31.0–31.5 FPS`；补充支撑/摆动相校准后的复测约 `30.0–30.5 FPS`，平均绘制约 `32.7–33.3 ms`，其中姿态计算仍约 `0.01 ms`，没有形成新的渲染热点。完全长成后的前后层四肢复用每帧画笔，四肢阶段约 `6.1–6.3 ms`。最终代码连续运行 2 分钟时 GDI 为 `60–61`、USER 为 `18–19`、私有内存约 `30.07–30.51 MB`，未观察到持续增长。

Debug 可直接设置 `BESKTOP_ACTION_PREVIEW`。Release 必须先设置 `BESKTOP_ENABLE_DIAGNOSTICS=1`；无效动作名会安全回退为无预览，并记录一条不含桌面项信息的 Warning。`side_kick` 与已有三个动作使用同一入口，推荐同时设置 `BESKTOP_MAX_ACTORS=1` 和 `BESKTOP_ANIMATION_SPEED=0.5` 观察姿态。

转身使用独立的 `BESKTOP_TURN_PREVIEW=1`；推荐配合 `BESKTOP_MAX_ACTORS=1` 和 `BESKTOP_ANIMATION_SPEED=0.25` 或 `0.5`。若同时设置动作预览和转身预览，首演员优先执行转身预览。该变量同样由 `RuntimeOptions` 集中解析，普通 Release 未开启 diagnostics 时忽略。

加入 `side_kick` 后，同一参考环境的 Release x64 全量 47 演员连续运行 125 秒，稳定约 `30.5–31.5 FPS`，平均绘制约 `31.8–32.7 ms`，姿态阶段仍约 `0.01 ms`。GDI 为 `60–61`、USER 为 `18–19`、私有内存约 `30.32–30.85 MB`、句柄为 `399–410`，首尾均未持续增长。

阶段 A 已完成以下验证：x64 Debug、x64 Release、Win32 Release 的 GUI、Pack CLI 和动作逻辑测试均构建通过，三组 CTest 均能发现并通过 `besktop_action_tests`；0.5 倍速实际预览了三个动作，8 倍速跨 Contact 的一次性事件由纯逻辑测试覆盖。普通 Release 未开启诊断时会忽略预览及演员限制变量。参考环境全量 47 个演员稳定约 30.0–31.0 FPS，运行 125 秒期间 GDI 句柄保持 60、USER 句柄保持 18，私有内存约 30.81–30.95 MB。

第一轮动作库补齐属于快速实现与人工微调阶段，本轮只复用现有目录验证 x64 Release：`besktop`、`besktop_action_tests` 构建通过，直接测试与 CTest 均通过，普通 Release 启动后 Esc 退出码为 `0`。全量演员 27 秒短测的大多数稳定窗口为 `31.0–32.0 FPS`、平均绘制约 `31.3–32.2 ms`；出现过一个 `28 FPS / 35.0 ms` 的短窗口，但未持续跌出当前约 30 FPS 基线。本结果不替代阶段收口时的 x64/Win32、长期资源和兼容性矩阵，也不包含 Codex 对动作视觉是否自然的判断。

### 阶段 B：拳法、防守与打空

状态：**单演员动作数据已完成，双演员交互未开始**。

- 用固定的两个演员完成直拳—拨挡、直拳—侧闪、挥拳—打空、上勾拳—轻受击。

验收：攻击和防守接触时刻一致；低帧率或 8 倍动画速度测试不重复/遗漏事件。

### 阶段 C：腿法、转身与重击反馈

状态：**单演员动作数据已完成，双演员交互未开始**。

- 复用已完成的 `front_kick`、`side_kick`、`roundhouse_kick`、`spinning_back_kick` 单演员 clip。
- 复用已完成的 `heavy_stagger` 反馈 clip。
- 侧踢和前踢复用支撑脚锁定；回旋踢驱动髋部和支撑脚旋转。
- 转身后踢必须包含头肩/薄片预转、侧面对屏幕、踢击、回正四段，不把它简化成原地抬腿。

验收：脚长不拉伸、支撑脚不明显滑动、薄片正反面不镜像、重击后退不进入任务栏和屏幕外。

### 阶段 D：受控互动导演

- 新增轻量 `CombatDirector` 和 `CombatPair`。
- 从距离合适且没有冷却的演员中选择双方。
- 先靠近到动作需要的距离，再停止漫游并对齐朝向。
- 同时只允许少量交互对，其他图标继续自由漫游。
- 动作结束后设置随机冷却，避免同一批图标持续打架。
- 默认演出按固定随机种子保持可复现，同时保留个体差异。

验收：全量桌面不会瞬间聚成一团；至少能自然出现命中、格挡、闪避、打空和反击；关闭战斗导演时仍保持当前纯漫游行为。

### 阶段 E：第一轮完整回归与节奏打磨

- 调整准备、接触、回收和受击硬直时长。
- 限制重动作和旋转动作的同时触发数量。

## 建议代码边界

优先拆出独立模块，避免继续扩大单个 `icon_fight_scene.cpp`：

```text
src/besktop/animation/action_clip.h/.cpp
  ActionId、ActionPhase、ActionClip、ContactEvent、内置动作表

src/besktop/animation/action_player.h/.cpp
  ActionInstance、时间推进、阶段和事件消费

src/besktop/animation/combat_director.h/.cpp
  演员配对、动作选择、结果计划和冷却

src/besktop/animation/icon_fight_scene.*
  演员生命周期、漫游、动作系统集成和渲染入口
```

关键姿态采样和 IK 数学可以先放在 `action_clip.cpp`，但不得依赖 Win32 窗口或桌面采集对象，便于后续增加纯逻辑测试。

## 调试与可复现性

建议逐步增加以下只读诊断选项，全部由 `RuntimeOptions` 集中解析：

```text
BESKTOP_ACTION_PREVIEW=<action_id>
BESKTOP_COMBAT_PAIRS=<数量>
BESKTOP_ACTION_SEED=<整数>
```

Debug 可以直接使用；Release 必须先设置 `BESKTOP_ENABLE_DIAGNOSTICS=1`。普通 Release 默认不记录演员名称、路径、动作明细或逐帧事件。

动作预览应兼容现有：

- `BESKTOP_ANIMATION_SPEED`
- `BESKTOP_ANIMATION_OFFSET`
- `BESKTOP_FRAME_STATS`
- `BESKTOP_MAX_ACTORS`

## 性能约束

第一轮动作必须遵守 [RENDER_PERFORMANCE.md](RENDER_PERFORMANCE.md)：

- 参考环境全量 47 个演员稳定保持至少 30 FPS。
- 同环境稳定 FPS 和平均绘制耗时原则上不得回退超过 10%；若 30 FPS 底线更严格，以底线为准。
- 同时测量全量演员与 10 个演员，并覆盖动作接触高峰，不只测稳定漫游。
- 动作姿态、事件和命中数学每演员每帧只计算一次。
- 不通过减少默认演员、删除文字、降低动画速度或降低刷新目标掩盖回退。
- 至少运行 5 分钟，确认 GDI 句柄和私有内存稳定。

图标主体仍是当前最大渲染热点。第一轮动作应尽量保持每演员每帧一次图标主体绘制；额外残影、拖尾、多层闪白和粒子放到性能预算更充足的后续阶段。
